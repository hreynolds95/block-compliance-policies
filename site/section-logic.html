1. Data Pipeline Overview

Source

  • Snowflake table: LOGICGATE_SF.PUBLIC.POLICY_DASHBOARD
  • Role: LOGICGATE_SF__SNOWFLAKE__READ_ONLY
  • Warehouse: ADHOC__SMALL
  • Filter: DOCUMENT_STATUS = 'Active'

Data Flow

Snowflake  →  export CSV  →  import_from_logicgate.py  →  docs/*.md  →  generate_docs_data.py  →  docs-data.json  →  GitHub Pages

Import & Deduplication

scripts/import_from_logicgate.py processes the Snowflake CSV export and creates one Markdown stub per unique policy. Deduplication mirrors the Blockcell dashboard logic:

  • Deduplicate by PWF_RECORD_ID — one stub per unique record
  • Priority order: Published > New Request Draft in Progress > Retired
  • Step order: Active Version > Publication > all others
  • Tiebreaker: fewest null fields
  • Excludes 3 test records by name (EXAMPLE / Test Document)

Document IDs

PrefixDomain
CP-NNNConsumer Protection
EE-NNNEthics & Employee Conduct
FC-NNNFinancial Crimes
GOV-NNNGovernance

IDs are assigned sequentially per domain after deduplication and remain stable across re-imports (existing files are never overwritten).

docs-data.json Generation

scripts/generate_docs_data.py walks the docs/ directory and extracts YAML frontmatter from every stub into site/docs-data.json. The file includes a full UTC ISO timestamp (generated) and source label used by the data source badge on the dashboard.

Review Date Sync

scripts/sync_review_dates.py corrects next_review_date by matching docs to dashboard-data.json via PWF_RECORD_ID. See Section 3 for full logic.

3. Review Date Logic

Background

During the October 2025 migration to LogicGate, review dates were not captured for pre-existing policies. The initial import set next_review_date directly from DUE_DATE — the LogicGate workflow approval deadline — which had already passed for completed workflows, causing 136 false-positive overdue flags.

Correction (sync_review_dates.py)

The script matches each DMS doc to the dashboard record by logicgate_record_id → PWF_RECORD_ID and computes next_review_date as follows:

DUE_DATE_STATUSnext_review_date formula
Complete DATE_OF_FINAL_APPROVAL + 1 year — workflow finished, next cycle is annual from last approval
Current / Coming Due / Extended / Overdue Past Extension / Pending Review DUE_DATE — active workflow deadline is the correct target date
3 docs are unmatched to dashboard records: GOV-008, GOV-010 (record IDs not in current dashboard), and POL-003 (removed — no PWF_RECORD_ID). These retain their manually set review dates.

review_status Calculation

Computed at JSON generation time in generate_docs_data.py:

delta = (next_review_date - today).days
delta < 0    → "overdue"
delta <= 30  → "due-soon"
else         → "ok"
missing      → "unknown"

4. Extension Status

Policies with active or pending due date extensions in LogicGate are flagged with extension_status in their frontmatter. The source field is EXTENSION_LIFECYCLE_STATUS in the Snowflake table.

extension_statusLogicGate valueBadgeMeaning
approved Extension Approved Ext. Approved Due date extension granted; extended_due_date is the new deadline
in-progress Extension In Progress Ext. Pending Extension request submitted, awaiting approval

Filters

  • Search — free-text match on doc_id, title, owner, domain
  • Domain — populated dynamically from loaded documents
  • Status — published / draft / in-review / retired
  • Review State — overdue / due-soon / ok
  • Extensions — any active / approved only / in-progress only

Active filter dropdowns are highlighted with an info-blue tint. Clicking a KPI card sets the matching filter and highlights that card.

Sorting

All columns are sortable by clicking the column header. Default sort is by doc_id ascending. The table header row is sticky and remains visible while scrolling.

6. Audit Log

An immutable append-only log of every document change event, stored at audit/audit-log.jsonl (one JSON object per line). Collapsed by default on the dashboard.

Schema

{
  "timestamp": "2026-04-07T18:00:00Z",   // ISO 8601 UTC
  "event":     "document_approved",       // event type
  "doc_id":    "GOV-001",
  "title":     "Block Inc. Compliance Management System Policy",
  "version":   "1.0.0",
  "actor":     "github-actions[bot]",
  "run_url":   "https://github.com/..."   // link to CI run
}

How It's Written

The audit-log.yml GitHub Actions workflow triggers on every merge to main. It calls scripts/append_audit_entry.py, which compares the current commit's docs to the previous commit and appends one entry per changed document. Git tags (doc/DOC-ID/vX.Y.Z) are also created on each merge.

Event Types

EventTrigger
document_approvedA doc stub was added or its version/status changed
repository_initializedFirst commit establishing the DMS
