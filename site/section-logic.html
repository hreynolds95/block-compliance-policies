1. Data Pipeline Overview

Source

  • Snowflake table: LOGICGATE_SF.PUBLIC.POLICY_DASHBOARD
  • Role: LOGICGATE_SF__SNOWFLAKE__READ_ONLY
  • Warehouse: ADHOC__SMALL
  • Filter: DOCUMENT_STATUS = 'Active'

Data Flow

Snowflake  →  export CSV  →  import_from_logicgate.py  →  docs/*.md  →  generate_docs_data.py  →  docs-data.json  →  GitHub Pages

Import & Deduplication

scripts/import_from_logicgate.py processes the Snowflake CSV export and creates one Markdown stub per unique policy. Deduplication mirrors the Blockcell dashboard logic:

  • Deduplicate by PWF_RECORD_ID — one stub per unique record
  • Priority order: Published > New Request Draft in Progress > Retired
  • Step order: Active Version > Publication > all others
  • Tiebreaker: fewest null fields
  • Excludes 3 test records by name (EXAMPLE / Test Document)

Document IDs

PrefixDomain
CP-NNNConsumer Protection
EE-NNNEthics & Employee Conduct
FC-NNNFinancial Crimes
GOV-NNNGovernance

IDs are assigned sequentially per domain after deduplication and remain stable across re-imports (existing files are never overwritten).

docs-data.json Generation

scripts/generate_docs_data.py walks the docs/ directory and extracts YAML frontmatter from every stub into site/docs-data.json. The file includes a full UTC ISO timestamp (generated) and source label used by the data source badge on the dashboard.

Review Date Sync

scripts/sync_review_dates.py corrects next_review_date by matching docs to dashboard-data.json via PWF_RECORD_ID. See Section 3 for full logic.

2. KPI Cards (Metrics Dashboard)

Seven KPI cards appear at the top of the Metrics Dashboard. Six are clickable links that navigate to the Policy Library with a pre-applied filter via URL param. The Active card is display-only.

CardColorDefinitionURL param
Active green status === 'published' — all live published policies and standards display only
Intake blue status === 'draft' || status === 'in-review' — documents in the approval pipeline ?status=not-published
Overdue Review red review_status is overdue, pending-review, or overdue-past-extension — any doc past its review deadline ?review=overdue
Coming Due (90d) yellow review_status === 'due-soon' || review_status === 'extension-coming-due' — next review date within 90 days, or Snowflake explicitly reports DUE_DATE_STATUS = 'Coming Due' ?review=coming-due
Active Extensions purple extension_status === 'approved' || extension_status === 'in-progress' — sourced from EXTENSION_LIFECYCLE_STATUS in Snowflake due_items ?extension=active
Recently Published (30d) blue status === 'published' and effective_date >= today − 30 days — policies approved and published in the last 30 days ?recent=30
Retired gray status === 'retired' — documents formally retired in LogicGate (WORKFLOW_STATUS = 'Retired') ?status=retired
The ?recent=30 param is handled specially: it sets an in-memory recentDays state (not a select filter) and shows a dismissable chip — "Recently Published (30d) ×" — in the filter bar to indicate the active filter.

4. Review Date & Status Logic

sync_review_dates.py — Fields Written to Frontmatter

Matches each DMS doc to the dashboard record by logicgate_record_id → PWF_RECORD_ID. On each data refresh it writes four fields to doc frontmatter:

Frontmatter fieldSource field(s)Notes
next_review_date EXTENDED_DUEDATE (when extension approved) or DUE_DATE For Complete rows, next cycle is sourced from due_items; see below
due_date_status DUE_DATE_STATUS Current / Coming Due / Extended / Pending Review / Overdue / Complete / etc.
lifecycle_status TOLLGATE + CURRENT_STEP in due_items current / under-qc / in-approvals — published docs only
extension_status EXTENSION_LIFECYCLE_STATUS in due_items approved / in-progress — set when an active extension exists

Complete Row Enrichment

When a doc's main rows record has DUE_DATE_STATUS = 'Complete' (prior review cycle finished), the sync script overlays four fields from the best matching due_items entry for the next open cycle: DUE_DATE, DUE_DATE_STATUS, EXTENDED_DUEDATE, and EXTENSION_LIFECYCLE_STATUS. This is critical for docs whose extensions live in due_items but not the main rows record.

GOV-008 and GOV-010 are unmatched to dashboard records (record IDs not present in current Snowflake data). These retain their manually set review dates.

review_status Classification

Computed at JSON generation time in generate_docs_data.py. Snowflake's due_date_status is the authoritative source — date math only refines within each status bucket.

# Primary path: due_date_status is set and non-Complete
is_extension = due_date_status in ("Extended", "Extension Coming Due", "Overdue Past Extension")
delta = (next_review_date - today).days

if delta < 0:
    if is_extension  → "overdue-past-extension"   # past extended deadline
    elif Overdue     → "overdue"                  # >30 days past due, no extension
    else             → "pending-review"            # within 30-day grace period

elif delta <= 90 OR due_date_status == "Coming Due":
    if is_extension  → "extension-coming-due"
    else             → "due-soon"
    # Note: Snowflake's "Coming Due" is trusted directly even if date > 90 days
    # (Snowflake computes a longer window tied to board/committee cycles)

else → "ok"

# due_date_status == "Complete" → "ok"
# No due_date_status (fallback date-math path):
#   delta < -30 → "overdue" | delta < 0 → "pending-review"
#   delta <= 90 → "due-soon" | else → "ok" | no date → "unknown"

Coming Due Window

The coming-due window is 90 days. Additionally, any doc where Snowflake's DUE_DATE_STATUS is explicitly 'Coming Due' is always classified as due-soon regardless of days remaining — Snowflake may flag docs beyond 90 days based on board or committee meeting schedules.

  • Domain — populated dynamically from loaded documents
  • Status — published / draft / in-review / retired / intake (draft + in-review)

  • Extensions — any active / approved only / in-progress only

Active filter dropdowns are highlighted. All filters (including Recently Published) are cleared by the Clear button or the chip's × button.

Sorting

All columns are sortable by clicking the column header. Default sort is by doc_id ascending. The table header row is sticky and remains visible while scrolling.

7. Audit Log

An immutable append-only log of every document change event, stored at audit/audit-log.jsonl (one JSON object per line). Collapsed by default on the dashboard.

Schema

{
  "timestamp": "2026-04-07T18:00:00Z",   // ISO 8601 UTC
  "event":     "document_approved",       // event type
  "doc_id":    "GOV-001",
  "title":     "Block Inc. Compliance Management System Policy",
  "version":   "1.0.0",
  "actor":     "github-actions[bot]",
  "run_url":   "https://github.com/..."   // link to CI run
}

How It's Written

The audit-log.yml GitHub Actions workflow triggers on every merge to main. It calls scripts/append_audit_entry.py, which compares the current commit's docs to the previous commit and appends one entry per changed document. Git tags (doc/DOC-ID/vX.Y.Z) are also created on each merge.

Event Types

EventTrigger
document_approvedA doc stub was added or its version/status changed
repository_initializedFirst commit establishing the DMS
